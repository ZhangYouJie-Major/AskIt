# AskIt 下一阶段开发路线设计

## 1. 目标

基于当前仓库实现状态，整理 AskIt 下一阶段开发路线，输出一份可执行的优先级计划。

本设计聚焦两个问题：

- 哪些能力已经完成，哪些仍处于半成品或未开始状态
- 下一阶段应该按什么顺序推进，才能尽快把系统从“可演示”推进到“可交付”

## 2. 范围

### 2.1 In Scope

- 用户可感知业务功能盘点
- 技术基础设施与运维能力盘点
- 半成品与未闭环能力识别
- 代码质量与架构债务识别
- 按 `P0 / P1 / P2` 输出下一阶段任务清单

### 2.2 Out of Scope

- 本文不直接进入代码实现
- 本文不拆到具体函数级改动
- 本文不覆盖长期演进方向，如多租户、复杂工作流、插件化能力

## 3. 当前状态判断

当前项目并非从零开始，已经具备基础知识库系统雏形，但整体仍停留在“核心主链可部分演示、尚未稳定交付”的阶段。

### 3.1 已完成能力

- RBAC 权限体系
  - 用户、角色、权限模型已落地
  - 后端 API 已实现
  - 前端用户管理、角色管理页面已接通
- 部门管理
  - 部门管理 API 已实现
  - 前端部门管理页已落地
  - 用户列表支持按部门筛选
- 登录认证
  - 登录、注册、当前用户信息接口已存在
  - 前端已实现登录态持久化
- 基础 RAG 查询能力
  - 问答页已存在
  - 查询 API 与向量检索服务已接通
- 文档处理基础组件
  - 解析器、分块器、Embedding 相关基础代码与测试已存在

### 3.2 半成品能力

- 文档上传闭环未完成
  - 当前上传接口主要是写入数据库记录
  - 实际文件存储、解析、分块、向量化、入库链路未完整接通
- 文档删除闭环未完成
  - 删除数据库记录后未同步清理文件、分块、向量数据
- Celery 异步任务仍为占位实现
  - 文档处理任务、清理任务、重建索引任务尚未具备生产可用能力
- 管理后台总览页未正式接入
  - 页面已存在，但路由未接通，局部交互仍为占位
- 会话历史仅存在于当前前端内存
  - 后端已有会话模型，但未提供正式 API 与前端入口

### 3.3 未开始能力

- 系统监控最小版本
- 文档处理运维视图
- 会话中心或历史会话管理
- OCR 正式处理链路核实与补齐
- 联网增强检索正式产品化

### 3.4 技术债

- 用户列表存在潜在 N+1 查询
- 用户列表分页为假分页
- 调试日志偏多，生产可用性不足
- 存在未接路由与死代码风险
- README 与代码实际能力存在承诺漂移

## 4. 设计原则

下一阶段按以下原则推进：

- `KISS`
  - 优先打通最短可交付路径，不引入额外架构层
- `YAGNI`
  - 只补当前明确需要的最小能力，不提前设计复杂任务编排或高级后台系统
- `DRY`
  - 文档处理链路统一由单一任务流负责，不在上传接口、后台重试、索引重建中复制处理逻辑
- `SOLID`
  - 上传、解析、分块、向量化、状态回写、监控展示分层明确
  - 前端管理页与文档处理后端职责分离

## 5. 方案比较

### 方案 A：链路优先

先补文档处理主链路，再补异步任务与状态可视化，随后完善后台与监控，最后补产品完成度。

优点：

- 最快提升系统真实可用性
- 最符合当前项目“核心价值链未闭环”的现状
- 风险最低，便于阶段验收

缺点：

- 前端可见页面提升速度不如“后台优先”方案快

### 方案 B：后台优先

先补管理后台总览和文档管理界面，再回头补底层链路。

优点：

- 界面完成度上升更快

缺点：

- 容易形成“页面齐了，但能力不稳定”的假完成状态

### 方案 C：监控优先

先补监控、日志、运行状态，再补业务闭环。

优点：

- 适合已有线上负载、急需排障的项目

缺点：

- 不适合当前项目当前阶段
- 当前主要问题不是观测不到，而是核心能力本身未闭环

### 结论

推荐采用 `方案 A：链路优先`。

## 6. 下一阶段任务清单

### 6.1 P0：核心链路闭环

目标：让“上传文档后可稳定进入知识库并用于问答”成为真实能力。

#### 任务 1：补文档上传真实闭环

- 接入实际文件存储
- 统一文件路径生成与元数据落库
- 上传成功后触发异步处理任务
- 失败时避免数据库和文件存储残留脏数据

验收标准：

- 上传合法文件后可看到数据库记录、文件存储记录、处理任务启动
- 上传失败时无孤儿文件、无无效数据库记录

#### 任务 2：补解析、分块、向量化、入库正式流水线

- 根据文件类型选择解析器
- 标准化解析结果
- 生成分块并写入 `document_chunks`
- 调用 Embedding 服务生成向量
- 写入 Chroma 并同步更新文档状态字段

验收标准：

- 一个已上传文档最终可达 `completed`
- `chunk_count`、`vectorized`、`error_message` 字段准确
- 文档内容可以被问答检索命中

#### 任务 3：补文档删除闭环

- 删除原文件
- 删除关联分块
- 删除关联向量
- 删除数据库主记录

验收标准：

- 删除后列表消失
- 向量检索不再命中目标文档
- 无孤儿 chunk、孤儿 vector

#### 任务 4：补异步任务正式实现

- 将文档处理任务改为真实生产链路
- 补齐状态回写
- 补齐失败日志与异常处理
- 提供最小可用索引重建能力

验收标准：

- 任一步骤失败时状态回写准确
- 能定位失败原因
- 能手动执行最小版本重建索引

#### 任务 5：补文档处理状态可视化

- 文档列表展示真实状态
- 失败时展示失败原因
- 提供手动刷新或最小轮询

验收标准：

- 上传后用户能观察到状态流转
- 失败时用户可以知道失败原因

### 6.2 P1：后台可管理、系统可观测

目标：让管理员能使用后台排查问题、追踪文档处理情况，并判断系统是否健康。

#### 任务 6：重构并接入管理后台总览页

- 正式接入后台首页
- 展示核心统计
- 展示最近文档处理记录

验收标准：

- 页面存在正式入口
- 所有统计来自真实 API

#### 任务 7：补文档管理后台能力

- 文档详情
- 处理状态详情
- 失败原因展示
- 重新处理入口

验收标准：

- 管理员可以对失败文档进行定位和重试

#### 任务 8：补系统监控最小版本

- 健康检查增强
- Celery 队列状态
- 向量库统计
- 文件存储状态
- 最近失败任务统计

验收标准：

- 可以快速判断服务、队列、向量库、文件存储哪一层出现问题

#### 任务 9：补后台运行概览

- 最近上传文档
- 最近失败任务
- 各状态任务数量

验收标准：

- 管理员可以在单页内获得运行概况

### 6.3 P2：提升产品完成度

目标：在主链路稳定后，提升终端用户体验和产品成熟度。

#### 任务 10：补会话持久化

- 新建会话
- 保存消息
- 获取历史会话
- 获取会话消息

验收标准：

- 刷新后历史仍可恢复
- 会话可以继续追问

#### 任务 11：补历史会话入口

- 会话列表
- 会话切换
- 历史消息引用来源查看

验收标准：

- 用户可以恢复旧会话继续工作

#### 任务 12：核实并补真实 OCR 链路

- 明确现有 OCR 能力是否已真正接入
- 若未接入，则补最小可用版本

验收标准：

- 至少一种图片型文档完成 OCR 后可以进入检索链路

#### 任务 13：核实并补联网增强检索

- 明确联网检索是否保留为正式能力
- 若保留，则实现最小产品化版本
- 若不保留，则下调 README 与对外说明

验收标准：

- 对 README 声明形成一致口径

## 7. 技术债处理建议

这些问题不单独作为里程碑，但建议在相关任务实施时顺手修复。

- 修正用户列表假分页
- 消除用户列表 N+1 查询
- 收敛调试日志输出
- 清理未接路由和死代码
- 修订 README 与系统真实状态不一致的问题

## 8. 推荐实施顺序

1. 文档上传真实闭环
2. 文档处理正式流水线
3. 文档删除闭环
4. 异步任务与状态回写
5. 前端状态可视化
6. 管理后台总览页正式接入
7. 文档管理后台增强
8. 系统监控最小版本
9. 会话持久化与历史会话
10. OCR 与联网增强能力核实后决定是否落地

## 9. 阶段验收标准

### P0 验收

- 上传文档后可进入知识库并用于问答
- 删除文档后无残留向量和分块脏数据
- 失败文档有明确错误信息

### P1 验收

- 管理员可通过后台判断系统运行状态
- 可定位失败任务并触发补救操作

### P2 验收

- 用户历史会话可恢复
- README 中声明的高级能力与实际状态一致

## 10. 风险与边界

- 当前最容易出现的问题不是功能不够多，而是半成品能力误判为已完成
- 在 P0 未完成前，不建议插入复杂新页面或新产品能力
- 若 OCR 或联网增强缺少可复用基础，应优先下调口径，而不是仓促补“伪实现”

## 11. 结论

下一阶段应优先解决“知识库主链路未闭环”的问题，而不是优先扩展表层功能数量。

推荐策略：

- 先做 `P0`，把系统从“可演示”推进到“可稳定运行”
- 再做 `P1`，让管理员可以使用后台管理和排障
- 最后做 `P2`，提升产品完成度和用户体验

这是当前阶段最符合系统现状、实现成本和交付价值的路线。
